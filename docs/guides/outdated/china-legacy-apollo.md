# 旧版中国版Apollo网络服流程<!-- md:flag china -->

这是一篇面向旧版中国版Apollo网络服项目维护的归档教程。它对应旧版MC Studio与Apollo文档中的“入驻开服→开发调试→提审发布→运营维护”链路，主要用于接手历史项目时快速定位流程节点。

/// warning | 仅用于旧版中国版生态
本页中的角色划分、平台审核、运维入口和工具按钮来自旧版中国版资料，不适用于国际版BDS、Realm或国际版脚本API工作流。
///

## 先判断是否需要本页

如果项目中出现以下痕迹，通常应先按本页梳理：

- 目录和配置中出现大厅服、游戏服、控制服、功能服等多角色服务。
- 项目由旧版MC Studio网络服模板创建。
- 维护任务涉及测试服提审、正式服申请、平台上架、运营指令或官方插件。

如果你是从零开始做新国际版服务端项目，本页不应作为主线教程。

## 旧版流程总览

旧版Apollo常见维护路径可压缩为六步：

1. 确认网络服入驻状态和开发机可用性。
2. 在MC Studio中创建或导入网络服工程。
3. 配置大厅服、游戏服、版本、地图与数据库。
4. 在开发机部署并执行联机调试。
5. 提交测试服审核并处理反馈，再申请正式服。
6. 上线后接入日志、监控和运营工具持续维护。

## 第一步：入驻与开发机

旧版资料中，网络服开发前需要先完成平台入驻流程，然后申请并连接开发机。常见准备项包括：

- 开发者身份与项目权限；
- 开发机连接信息；
- SSH密钥与目录访问；
- 多账号协作权限分工。

如果开发机不可用，后续部署、调试、提审都会被阻断，建议把该检查放在首位。

## 第二步：创建或导入工程

旧版MC Studio支持两种入口：

- **模板新建**：空白模板、简易网络服模板等。
- **本地导入**：导入已有部署脚本与工程目录。

维护旧工程时，建议先确认类型名、目录名、Mod目录路径和部署配置是否仍匹配当前机器环境，再执行部署。

## 第三步：服务与资源配置

在旧版配置阶段，常见重点包括：

- 大厅服与游戏服角色分配；
- 版本选择与升级策略；
- 地图保存策略与地图体积控制；
- 数据库连接（如MySQL、MongoDB、Redis）；
- 服务器Mod与插件装配。

旧版资料强调，玩家登录通常依赖`lobby`类型服务；若未正确配置大厅角色，玩家可能无法正常进入网络服。

## 第四步：开发调试

旧版流程通常包含以下调试环节：

- 本地或开发机部署；
- 控制台日志排查；
- 多账号联机验证；
- 热更新与回滚验证；
- 服务器Mod加载顺序与性能采样。

复杂问题可先拆分为“单服可复现”与“跨服可复现”两类，再分别定位到游戏服逻辑或跨服通信链路。

## 第五步：提审、申请正式服与上架

旧版平台链路通常是：

1. 进入未审核列表管理目标网络服。
2. 更新服务器基础信息并提交测试服审核。
3. 审核通过后申请正式服并执行部署演练。
4. 满足平台要求后正式上架。

提审期间应确保测试服可访问，避免因环境不可用导致审核结论失真。

## 第六步：上线后维护

旧版资料中的运维重点通常包括：

- 日志体系接入与检索；
- 监控报警与可视化面板；
- 常见问题与实用技巧沉淀；
- 运营指令与活动插件配置；
- 性能回归与容量规划。

对历史项目而言，先保运行、后做迁移通常比“一次性重构”更稳妥。
